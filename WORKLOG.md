## Unified Source DOM Implementation Plan

- Phase 1 - Source snapshot foundation: introduce a single `TherionSourceDocument`/future `TherionSourceDom` snapshot API that preserves physical lines, comments, blank lines, indentation, newline style, encoding metadata, source file type, absolute offsets, line/column token spans, and document revision identity; keep `TherionSourceText`, `TherionDocumentParser::parseSourceDocument()`, and current `TherionSourceDocument` behavior as the compatibility bridge while adding focused tests for quoted strings, comments, blank lines, continuation lines, bracket groups, and mixed newline round trips.
- Phase 2 - Logical command layer: add a lossless logical-command projection over the snapshot that groups continuation lines, classifies directive rows versus block body rows, records block open/close pairs, represents ambiguous inline/block directives such as `source` and `revise` without guessing, and exposes command name, positional arguments, option entries, raw value tokens, logical value counts, and precise source ranges through one core API.
- Phase 3 - Catalog-aware command metadata: make catalog-derived command domains, option aliases, option arity, allowed values, argument requirements, and document-type applicability available to the logical-command layer through injected metadata; keep `.th`, `.th2`, and `thconfig` domain differences explicit, and avoid app-side hardcoded exceptions except as short-lived regression-protected compatibility shims.
- Phase 4 - Validator migration: keep `TherionSourceValidator` consuming the logical-command projection for catalog-backed command validation with physical source-range mapping for token diagnostics; preserve conservative severity policy, token-level ranges, safe-fix metadata, data/reference row suppression, thconfig uncertainty handling, and project-wide validation grouping; keep regression tests for real-world false positives already seen (`point ... label -text`, `export -o`, `survey -title ... \`, bracketed `-scale`, one-line `revise`, map references, centerline data rows).
- Phase 5 - Raw highlighter/help/completion migration: replace remaining raw highlighter invalid-token heuristics with validator diagnostics and shared token categories; make context help, completion, tooltip text, and underline ranges consume the same snapshot/logical-command/catalog data, keyed by document revision, so cursor hover/caret differences cannot show stale or conflicting diagnostics.
- Phase 6 - Blocks projection migration: make the Blocks canvas read cards, nesting, object-reference rows, fixed root `encoding`, centerline data bodies, and command details from the shared DOM projection; route Blocks edits through source-range transactions against DOM-owned ranges and keep existing undo/redo, auto-commit, option editing, and data-row editing tests passing.
- Phase 7 - Map/TH2 projection migration: make scraps, points, lines, areas, background metadata, area references, line-point rows, standalone line directives, Bezier geometry, and Smart Area source insertions read from the shared DOM projection; keep map source mutations atomic through `TextEditorSourceTransactionController` or its successor and add focused round-trip tests for geometry rewrite, area references, line-point standalone rows, and background metadata preservation.
- Phase 8 - Structure/project index migration: make the Structure sidebar, project validation scanner, search navigation, namespace/reference resolution, root-config inference, and diagnostics grouping consume cached DOM snapshots for project files; implement asynchronous cancellable parsing/indexing keyed by file identity plus revision, and keep unsaved in-memory document contents participating in project validation/search.
- Phase 9 - Transaction and cache model: centralize source mutations behind a single document source-change service with one logical undo label, revision check, source replacement, dirty-state update, projection invalidation, cursor/selection restoration, and post-change diagnostic refresh; cache DOM snapshots and projections by revision so cursor movement, hover, theme changes, and inspector selection do not reparse whole documents.
- Phase 10 - Legacy removal gates: after each consumer migrates, remove the corresponding editor-local tokenizer, option parser, numeric classifier, block classifier, source-range heuristic, or serializer wrapper; keep `StructureConstraintsTest` and focused architecture checks preventing new UI-side parsing/serialization paths from appearing in `src/app/**`, QGraphicsItems, dialogs, or inspector widgets.
- Verification gates: every phase shall include focused unit tests for parser/projection behavior, round-trip preservation tests for representative `.th`, `.th2`, and `thconfig` snippets, source-range tests for diagnostics and edits, undo/redo tests for user-visible mutations, and at least one release-build regression pass covering `CommandOptionParserTest`, `TherionSourceValidatorTest`, `ProjectValidationScannerTest`, `TextEditorCompletionHighlightTest`, `StructureConstraintsTest`, and the affected Map/Blocks tests.
