# Localization

This document describes how Therion Studio localization is built, updated, reviewed, and packaged.

## Scope

Therion Studio localizes the application UI, not Therion source language semantics.

Translate:

- menus, toolbars, dialogs, alerts, empty states, inspector labels, search controls, console labels, and help-panel chrome
- user-facing status messages and validation errors
- explanatory prose around commands, when it is clearly UI/help text

Do not translate:

- Therion commands such as `survey`, `centreline`, `scrap`, `point`, `line`, `area`, `map`, `data`, `endline`, and related closing directives
- Therion options such as `-id`, `-name`, `-projection`, `-scale`, `-orientation`, `-close`, `-reverse`, and `-subtype`
- accepted values, argument tokens, file-format keywords, command syntax examples, and serialized document content
- generated or user-entered file paths, object IDs, project names, station names, labels, and editor text

Localized UI may describe these concepts in Czech or Slovak, but canonical Therion spelling must remain visible where it represents file syntax.

## Current Languages

The source language is English.

Bundled application translations:

- Czech: `translations/therion_studio_cs.ts`
- Slovak: `translations/therion_studio_sk.ts`

The initial language set is advertised in:

- `CMakeLists.txt`, through `THERION_STUDIO_TRANSLATION_FILES`
- `src/platform/ApplicationStartupBootstrap.cpp`, through startup locale selection and translator loading
- `src/app/MainWindowSettingsDialog.cpp`, through the user-facing language picker
- `cmake/macos/Info.plist.in`, through `CFBundleLocalizations`

When adding a language, update all of these locations in the same change.

## Required Tools

Install Qt Linguist tools for the Qt version used by the build.

Common tools:

- `lupdate`: extracts source strings into `.ts` catalogs
- `linguist`: GUI editor for `.ts` catalogs
- `lrelease`: compiles `.ts` catalogs into `.qm` catalogs
- CMake targets generated by `qt_add_translations`: `update_translations` and `release_translations`

Platform package notes:

- macOS Homebrew builds should depend on both `qt` and `qttranslations`.
- Ubuntu/Debian development installs should include `qt6-tools-dev-tools` in addition to the Qt Widgets/Svg development packages.
- Windows developer machines should install Qt with the Linguist Tools component matching the compiler toolchain.

## String Markup Rules

Use normal Qt translation APIs for every user-facing literal.

For `QObject` subclasses:

```cpp
setWindowTitle(tr("Settings"));
```

For non-`QObject` controllers and services, prefer an explicit context:

```cpp
QCoreApplication::translate("TherionStudio::MapEditorObjectMovePlanner",
                            "Cannot move a map object in an empty document.");
```

Avoid custom `tr()` helpers in non-`QObject` classes unless they are known to extract and resolve under the same context. `lupdate` may warn that such classes lack `Q_OBJECT`; those warnings are harmless for existing code that resolves correctly, but new code should use explicit `QCoreApplication::translate(...)` to avoid context drift.

Keep placeholders stable:

- preserve `%1`, `%2`, and other Qt placeholders exactly
- preserve markdown links, code spans, and HTML tags
- preserve keyboard shortcut names when they refer to physical keys or platform conventions
- preserve command syntax inside `<code>...</code>` or markdown backticks

Example:

```xml
<source>The selected line is not a writable point command.</source>
<translation>Vybraný řádek není zapisovatelný příkaz `point`.</translation>
```

The prose is translated, but `point` remains canonical.

## Update Workflow

Start from a configured build tree:

```sh
cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug
```

Extract source strings:

```sh
cmake --build build --target update_translations
```

Edit translations:

```sh
linguist translations/therion_studio_cs.ts
linguist translations/therion_studio_sk.ts
```

Compile runtime catalogs:

```sh
cmake --build build --target release_translations
```

Build the application:

```sh
cmake --build build --target TherionStudio --parallel
```

Run tests:

```sh
ctest --test-dir build --output-on-failure --timeout 300
```

## Review Checklist

Before committing localization changes:

- `cmake --build build --target update_translations` finds no unexpected new strings
- `cmake --build build --target release_translations` reports zero unfinished translations for bundled languages
- `rg 'type="unfinished"' translations/therion_studio_*.ts` returns no output
- placeholders in source and translation match
- command syntax, accepted values, option names, and examples remain canonical Therion text
- application builds successfully
- relevant tests pass

A quick placeholder sanity check:

```sh
python3 - <<'PY'
import re
import xml.etree.ElementTree as ET

for path in ["translations/therion_studio_cs.ts", "translations/therion_studio_sk.ts"]:
    root = ET.parse(path).getroot()
    for context in root.findall("context"):
        name = context.findtext("name") or ""
        for message in context.findall("message"):
            source = "".join(message.find("source").itertext())
            translation = "".join(message.find("translation").itertext())
            source_args = sorted(set(re.findall(r"%\d+", source)))
            translation_args = sorted(set(re.findall(r"%\d+", translation)))
            if source_args != translation_args:
                raise SystemExit(f"{path} [{name}] placeholder mismatch: {source!r}")
print("translation placeholders ok")
PY
```

## Runtime Loading

Startup localization is handled in `src/platform/ApplicationStartupBootstrap.cpp`.

The startup sequence:

1. Resolves the preferred application locale from the platform override and session setting.
2. Loads Qt standard translation catalogs (`qtbase`, then `qt`) from `QLibraryInfo::TranslationsPath`.
3. Loads the bundled application catalog `therion_studio_<language>.qm` from the Qt resource prefix `:/i18n`.

English is the source and fallback language. Unsupported languages fall back to English.

Language changes currently take effect on next launch.

## Settings and Platform Overrides

The Settings dialog exposes:

- `System Default`
- `English`
- `Czech`
- `Slovak`

`System Default` means the application follows the operating system locale when a bundled translation is available.

On macOS, Therion Studio also writes the native per-app language override through the app-domain `AppleLanguages` preference. Selecting `System Default` clears that override so macOS controls the language again.

On Windows and Linux, the language override is stored in the application session/settings store and read at startup.

## Platform Packaging Notes

### macOS

The app bundle declares supported localizations in `cmake/macos/Info.plist.in` via `CFBundleLocalizations`.

Homebrew formula dependencies should include:

- `qt`
- `qttranslations`

`qttranslations` is needed so standard Qt and platform-provided labels, including native menu items such as `Preferences...`, can participate in localization when Qt provides the matching catalog.

### Windows

The Windows installer must include:

- bundled application `.qm` resources compiled into the executable resource system
- Qt runtime libraries and plugins deployed next to `bin/TherionStudio.exe`
- Qt translation catalogs when standard Qt dialog/menu localization is required at runtime

Keep the Qt version used by `lrelease`, the deployed Qt runtime, and the compiler toolchain aligned.

### Linux

The `.deb` package relies on distro Qt packages. The package dependency set should include the Qt runtime translation package when standard Qt strings are expected to localize.

The AppImage uses a staged AppDir. Ensure Qt translation catalogs needed by standard Qt UI strings are present in the AppDir if the target distro Qt deployment helper does not copy them automatically. Application translations are embedded through the `:/i18n` resource prefix.

## Adding A New Language

1. Add a `.ts` file under `translations/`, for example `translations/therion_studio_de.ts`.
2. Add it to `THERION_STUDIO_TRANSLATION_FILES` in `CMakeLists.txt`.
3. Add the language to startup language normalization/loading if it is user-selectable.
4. Add the language to `MainWindowSettingsDialog`.
5. Add the language to `CFBundleLocalizations` in `cmake/macos/Info.plist.in`.
6. Run `cmake --build build --target update_translations`.
7. Translate with Qt Linguist.
8. Run `release_translations`, build, tests, and the review checklist.
9. Update `docs/USER_MANUAL.md`, `QtReimplementationSpecification.md`, and release/packaging docs if the supported language set changes.

Do not add a language to the Settings dialog until its application catalog is complete enough for normal use.

## Common Pitfalls

- Translating Therion commands or options breaks user understanding of file syntax.
- Translating accepted values can make contextual help disagree with actual parser/serializer behavior.
- Dropping `%1` placeholders causes runtime message formatting bugs.
- Using different translation contexts for extraction and runtime lookup produces English fallbacks even when the `.ts` file contains a translation.
- Relying only on application `.qm` files does not localize Qt-provided standard buttons or platform menus; Qt translation catalogs must also be installed and found at runtime.
- Immediate live language switching is not currently required; changing language should be treated as a next-launch setting unless the architecture is explicitly extended.
